## 🧭 Week 11-12（Phase 3: UniversalRunner + LLM Recovery）

> **目标**: 完全迁移到目标架构（UniversalRunner + LLM 驱动恢复）

### A. UniversalRunner MVP（统一执行器） 🔄 进行中

**改动**：
- 新增 `backend/task/universal_runner.py`，统一 tick-loop 执行逻辑
- 兼容 `mine/goto/give/craft/scan/mine_tree` 等原子动作

**完成标准**：
- 一个 Runner 可执行“采集/导航/交付/合成”四类任务
- 不依赖 `GatherRunner/LinearPlanRunner`

**验收步骤**：
1. 编写/调整 `backend/tests/test_task_executor.py`，覆盖 4 类任务
2. 本地运行测试用例通过
3. 手动验证：`砍树 -> 合成木板 -> 交给玩家` 可闭环

**✅ 已完成的修复** (2026-01-04):
- [x] 多步意图提前终止: 新增 `_is_pure_single_step_task()` 判断，复合任务依赖 LLM done=true
- [x] 强制锚定主人: 新增 `_should_anchor_to_owner()`，仅明确要求时锚定
- [x] Craft 参数归一化: `_normalize_step()` 支持 craft 动作 KB 解析
- [x] RecoveryCoordinator 注入: `registry.py` 正确初始化并传入恢复协调器
- [x] L1 微重试: 实现 `RETRY_SAME` 内部循环，跳过 LLM 调用
- [x] 砍树锚定逻辑修正: 完善 `_resolve_search_center`，支持 LLM 指定/锚定/自由位置
- [x] Inventory Delta 优化: 改为辅助信息 (`bot_state`) 而非直接完成判据
- [x] 测试覆盖: 新增 `test_universal_runner.py` 覆盖 3 大核心场景 (10/10 通过)

### C. LLM 驱动恢复（替换硬编码 Recovery）

**改动**：
- 移除 RecoveryCoordinator 的硬恢复路径
- 新增 LLM Recovery Prompt 或 `planner.recover()` 接口
- 失败上下文（error_code + last_result + bot_state）回传 LLM

**完成标准**：
- 失败时由 LLM 生成恢复动作或澄清问题

**验收步骤**：
1. 模拟 `PATH_NOT_FOUND/NO_TOOL`，LLM 返回恢复动作
2. 不再出现 RecoveryCoordinator 的 L1/L2/L3/L4 分支日志

### 核心定律：不要“修改”历史节点，而是“追加”新状态（Append, Not Rewrite）

在你的直觉里，你可能会想：“计划全乱了，我得赶紧跑回 LangGraph 里，把那个存着旧计划的 State（状态）给删掉或者改掉，换成新的。”

**错！绝对不能改历史！** 
LangGraph 加上你刚装的 `SqliteSaver`（持久化检查点）的最大魅力就在于：**它是一本不可篡改的日记本**。

*   **为什么要留着旧的失败计划？** 
    因为大模型（LLM）需要**吸取教训**！如果你把被炸死的历史抹掉了，直接塞一个新环境给大模型，它会很懵：“我干嘛在这？主人刚才让我干啥来着？”
    你需要让 LLM 看到完整的上下文：“我本来在执行主人的命令挖树，进行到第二步时被苦力怕炸死了，我现在空手在重生点。” 这样，LLM 生成的下一句话（Bark）才会极其生动：“呜呜呜，主人对不起，苦力怕把我炸回老家了喵，东西全掉了，我这就跑回去捡喵！”

### 面对怪物袭击/死亡的完整梳理流程（教科书级异常处理）

现在，让我们模拟一次真实的“苦力怕爆炸”灾难，看看你的三个组件（LangGraph大脑、执行器身体、预取嘴巴）是如何完美配合的：

#### 阶段 1：灾难发生（物理层崩溃）
女仆正在执行第 2 步（做木板），同时后台正在美滋滋地生成第 3 步的台词。
**Boom！苦力怕炸了，女仆死亡并重生。**
Mineflayer（Node.js 端）立刻报错：“动作执行失败，实体死亡/目标丢失”。

#### 阶段 2：执行层紧急避险（断臂求生）
你的 `task_job_runner.py` 捕获到了这个物理报错。它需要做三件事：
1. **掐死幻觉**：立刻调用 `prefetch_task.cancel()`，把正在后台生成的第 3 步开心台词果断杀掉（不能死了还笑嘻嘻的）。
2. **清空旧计划**：当前队列里的后续步骤 `[做工作台, 做木镐]` 全部作废，直接丢弃。
3. **搜集现场证据**：重新调用 JS 获取一份**最新的环境快照**（此时的快照会显示：背包为空、血量满、坐标在重生点、周围没有树）。

#### 阶段 3：唤醒 LangGraph 进行“灾后重建”（重规）
**这是你最关心的部分！如何重新编排 State？**

我们**不在**旧的图节点里打转，而是拿着刚才搜集的“现场证据”，**向 LangGraph 发起一次全新的 Invoke（调用）！**

但这次调用，我们要给 LangGraph 传一个新的 State 更新（追加信息）：
```json
{
  "user_input": "（保持最初主人的命令：去挖石头）",
  "last_failure_reason": "执行 'craft(oak_planks)' 时被苦力怕炸死，已重生。",
  "env_snapshot": "（全新的空手重生点快照）"
}
```

因为你开启了 LangGraph 的 `thread_id`（线程记忆），LangGraph 读取这个请求时，会自动把之前的对话和旧计划加载进来，然后路由（Router）看到 `last_failure_reason`，就会把流程**强行导向** `TaskPlanner` 节点。

#### 阶段 4：大模型重新思考
LLM 看到完整的日记本：
> *过去*：主人让我挖石头。我计划去挖树。
> *过去*：我挖到了树，但在做木板时被炸死了。
> *现在*：我在重生点，没东西。

于是，`TaskPlanner` 产出了全新的计划和开场白：
*   **新开场白（Opening Bark）**：“呜呜呜吓死我了喵！好痛！主人稍等，我这就顺着原路跑回去把掉落的木头捡回来喵！”
*   **新计划（New Plan）**：`[move_to(死亡坐标), pick_up(木头), craft(木板), ...]`

这个新计划再次被送进执行队列，一切又像什么都没发生过一样，重新流转起来！

### 总结你的架构：为什么它如此强悍？

你不需要去“修改” LangGraph 的底层节点，因为你现在的架构天然支持这种打断：

*   **平时**：LangGraph 闭眼休息，执行层（队列）一条路走到黑，预取层疯狂说俏皮话。
*   **遇灾**：队列清空，预取取消，收集新快照 ➔ **把新快照作为新的 Message/Event 追加进原有的 `thread_id` 里，再次调用 LangGraph** ➔ LangGraph 生成新计划 ➔ 队列重新启动。

这就是为什么 AI 助手之前跟你强调 **“只有物理失败时，才切到恢复/重规”** 的深层逻辑！
